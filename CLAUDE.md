# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A lightweight SQLite3 ORM built on Python's stdlib `sqlite3`, with `pydantic` for schema/validation. Only runtime deps are `pydantic>=2.6` and `typing_extensions`. Supports Python 3.8–3.14, so most modules carry `sys.version_info` guards and `from __future__ import annotations` — preserve this when editing.

## Commands

Tooling is `uv` (see `uv.lock`); tests need `src` on the path, which `pyproject.toml` (`pythonpath = ["src"]`) handles.

- Run tests: `uv run pytest`
- With coverage (as CI does): `uv run coverage run -m pytest && uv run coverage report`
- Single test: `uv run pytest tests/test__table_spec.py::TestSomething::test_x`
- Lint / format: `uv run ruff check --fix` and `uv run ruff format`
- All pre-commit hooks: `pre-commit run --all-files`

`asyncio_mode = "auto"` — async tests need no `@pytest.mark.asyncio`. The `tests/test_e2e/` suite generates 120k rows (`TEST_ENTRY_NUM` in its conftest) and is slow; prefer targeted unit tests (`tests/test__*.py`) during iteration.

## Architecture

Three layers, each building on the one below:

1. **`TableSpec`** (`_table_spec.py`) — a `pydantic.BaseModel` subclass that *is* the table schema. Beyond holding row data, its `table_*_stmt` classmethods (`table_create_stmt`, `table_insert_stmt`, `table_select_stmt`, …) are pure SQL-string generators (all `@lru_cache`d, so their args must stay hashable tuples). It also provides `table_row_factory` (raw sqlite row → validated instance) and `table_serialize_mapping` (app-type mapping → DB-ready dict via pydantic).
2. **`ORMBase[T]`** (`_orm/_base.py`) — binds a `TableSpec` to a `sqlite3.Connection` and exposes CRUD (`orm_insert_*`, `orm_select_*`, `orm_update_*`, `orm_delete_*`, `orm_bootstrap_db`). Each method delegates statement construction to the `TableSpec`, then executes. Column values pass through `table_serialize_mapping` before binding.
3. **Pool layers** (`_orm/_pool.py`) — `ORMThreadPoolBase` and `AsyncORMBase` (experimental). They do **not** reimplement CRUD: methods are generated by wrapping the corresponding `ORMBase` method with `_wrap_with_thread_ctx` / `_wrap_with_async_ctx`. Each worker thread holds its own thread-scoped `ORMBase` (one connection per thread — SQLite connections aren't shareable across threads); generator results are bridged back over a bounded `queue.Queue`.

Key non-obvious mechanisms:

- **`ORMBase[MyTable]` is not just typing.** `__class_getitem__` (`parameterized_class_getitem` in `_orm/_utils.py`) creates and caches a real subclass with `orm_table_spec` set to `MyTable`. Subclassing that and setting `orm_bootstrap_table_name` is the normal way to define a concrete ORM (see `tests/sample_db/orm.py`).
- **Columns are typed via `Annotated` metadata.** `TypeAffinityRepr` maps a Python type → SQLite type affinity (`_utils.py::map_type`, handling `Optional`, `Literal`, `IntEnum`/`StrEnum`); `ConstrainRepr` composes constraint clauses (`NOT NULL`, `CHECK`, `DEFAULT`, …). `table_dump_column` reads these off each field to emit the column DDL.
- **row_factory selection** is driven by `RowFactorySpecifier` (a string like `"table_spec"` / `"table_spec_no_validation"`, a callable, or `None`); default returns validated `TableSpec` instances. Pass `"table_spec_no_validation"` to skip pydantic validation for speed.
- **datetime custom types** (`_types.py`): `DatetimeUnixTimestamp` (REAL), `DatetimeUnixTimestampInt` (INTEGER), `DatetimeISO8601` (TEXT) — pydantic `Annotated` aliases with serializer/validator baked in.
- **Custom SQL escape hatches:** most ORM methods accept a `_stmt=` (bypass generated statement) and where-generation supports raw `where_stmt=` + `_extra_params=`. For bulk updates use `orm_update_entries_many` (executemany), not `orm_update_entries` in a loop.

## Layout & conventions

- Underscore-prefixed modules (`_table_spec.py`, `_orm/`, `_utils.py`, `_sqlite_spec.py`, `_types.py`) are internal; `src/simple_sqlite3_orm/__init__.py` re-exports the public API from them. Import public names from the package root.
- Two public surfaces: the package root (core types) and **`simple_sqlite3_orm.utils`** — connection/PRAGMA tuning (`enable_wal_mode`, `enable_mmap`, `optimize_db`), introspection (`lookup_table`, `check_db_integrity`, `check_pragma_compile_time_options`), and DDL helpers (`gen_check_constrain`, `concatenate_condition`, `wrap_value`). Don't confuse it with the internal `_utils.py`.
- Public method naming: `TableSpec` → `table_*`, ORM classes → `orm_*`. Follow this when adding APIs.
- Features gated on the runtime SQLite build/version (RETURNING ≥3.35, STRICT ≥3.37, UPDATE/DELETE `LIMIT`/`ORDER BY` require `SQLITE_ENABLE_UPDATE_DELETE_LIMIT`) are detected in `tests/conftest.py::SQLITE3_FEATURE_FLAGS`; guard related tests with those flags.
- `_version.py` is generated by hatch-vcs — never edit or commit it.
